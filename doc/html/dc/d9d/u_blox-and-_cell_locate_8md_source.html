11 # CellLocate
12 Some short description about the CellLocate feature:
13 
14 > This technology enables stand-alone location estimation based on surrounding mobile network information in conjunction with GNSS positioning data to improve positioning. For any given location with cellular network coverage (2G, 3G or
15 > 4G), a specific combination of network cells will be visible. The proprietary CellLocate feature allows u-blox cellular modules to report to the CellLocate server those cells which are visible at any specific location. This enables the server to estimate a coarse position based on previous observations from other modules reporting the same cell visibility pattern, and this position is reported back to the module. The estimated position is then output by the module to the host processor via its serial port.
16 
17 > Network cells are widely available in urban and rural environments enabling the CellLocate service to provide a position estimate virtually everywhere and under any conditions. The service is offered free of charge on a best effort basis and the performance depends on the density of network cells and database population. A self-learning mechanism is implemented allowing continuous database improvement and update to increase performance over time and prevent database ageing.

31 # u-blox Command Guide
32 This quick guide will explain what steps need to be taken to be able to connect the u-blox module to the network, connect it to the CellLocate servers to be able to request the location and how to send a simple text message.
33 This is to be used to jump-start software development.
34 
35 u-blox suggests best practices for the software handling the communication to their modules on page 11 of the [`AT Commands Examples App Note`](https://www.u-blox.com/sites/default/files/AT-CommandsExamples_AppNote_%28UBX-13001820%29.pdf).
36 
37 Since February 2020, there is also a [`GitHub`](https://github.com/u-blox/cellular) page by u-blox, where they are currently developing a cellular API in C for embedded platforms.
38 
39 Frequently used abbreviations in the u-blox documentation:
40 * DCE (Data Communications Equipment) or MT (Mobile Terminal) is the u-blox cellular module
41 * DTE (Data Terminal Equipment) is the Raspberry PI
42 * NVM (Non-volatile Memory)
43 * IMEI (International Mobile Equipment Identity)

52 Specifically, a typical exchange looks like this:
53 
54 | PI sends | SARA responds |
55 | ------------- |----------------------------------------|
56 | AT+<cmd>\r | <echo>\r |
57 | | <response>\r (optional - can be empty)|
58 | | <result code>\r |
59 
60 Example:
61 | PI sends | SARA responds |
62 | ------------- |----------------------------------------|
63 | AT+UPSND=0.8\r| AT+UPSND=0.8\r |
64 | | +UPSND: 0,8,0\r |
65 | | OK\r |
66 
67 ### Result code can be:
68 * OK
69 * ERROR
70 * ABORTED
71 Error codes can be more informative if set to verbose with the following command:
72 AT+CMEE=2
73 
74 ### Unsolicited Result Code (URC)
75 The commands can trigger an action that will eventually get a result.
76 From the commands application note:
77 
78 > An unsolicited result code (URC) is a string message (provided by the DCE) that is not a response to a previous AT command. It can be output, when enabled, at any time to inform the DTE of a specific event or status change. The URC can have the same name as the command that enables it (e.g. +CREG) or can be enabled by another command (e.g. unsolicited result code: +CMTI, command that enables it: +CNMI).
79 
80 This is used to get the location. The command will trigger a couple of server requests and eventually result in an answer that can be checked.
81 
82 ### Summary: Types of Commands
83 Generally speaking, there are three types of commands/interactions:
84 * A command sent by the PI that is only acknowledged by the SARA module: only has an echo and the status.
85 * A command sent by the PI that immediately gets a response (before the status). <br> If this command is not valid, or not understood, the SARA module will respond with an ERROR immediately, and no response will be sent. <br> A response can be more than just one line, depending on the command and the content (e.g. reading all text messages on the module will trigger a bunch of `+CMGL: ...` responses, each at least two lines long.)
86 * A command sent by the PI that triggers sending of (one or more) URCs (e.g. the get location command). This command is echoed and acknowledged by the SARA module. After the command has been executed, the result is put out as a URC.
87 
88 ### Reading/Setting Parameters
89 Many commands have parameters (often multiple) that they can set. This is the general structure:
90 * reading a parameter: `AT+<cmd>?\r`
91 * setting a parameter: `AT+<cmd>=<x>\r`
92 * requesting possible parameters: `AT+<cmd>=?\r`

94 ## Setting up the u-blox module
95 Before the module can be used, it needs to be set up. Some parts can be configured and stored in NVM
96 
97 Make sure the connected SIM card has already been activated and has a data plan.
98 The above mentioned structure is now omitted and the following structure is now used:
99  `<cmd>` --> `<response>`

101 
102 ### Entering the PIN:
103 If the PIN is activated (recommended), it has to be entered after startup.
104 
105 --> Enter the PIN:
106 `AT+CPIN="1234"\r`<br>
107 The command `AT+CPIN=?\r` does not return the PIN, but the status. This can be either `SIM PIN` or `READY`.
108 
109 ### Automatic network registration:
110 This should happen automatically, if not it can be forced with:
111 
112 Forces an attempt to select and register with the network operator: `AT+COPS=0`
113 
114 #### Check GSM
115 To check if everything is ok: `AT+COPS?` --> `+COPS: 0,0,"O2 - UK"`
116 If the first parameter is 0, the module is successfully registered with the GSM network.
117 
118 #### Check GPRS
119 `AT+CGATT?\r` --> `+CGATT: 1\r` GPRS attach status (1: attached)
120 
121 #### Check Packet-Switched Data
122 For this to work, some additional (one time only) setup is necessary. Afterwards, Packet-Switched (PS) network registration status can be checked:
123 `AT+UCGOPS?\r` --> `+UCGOPS: 0,0,"O2 - UK"\r`<br>
124 This does mean that an internet connection is available, but the **internet connection needs to be activated after every start-up**.
125 
126 ### PDP context activation:
127 
128 Check the status of the PDP connection profile (second digit: 8) associated to GPRS connection profile identifier "0" (first digit): <br>
129 `AT+UPSND=0,8\r` --> `+UPSND: 0,8,0\r`<br>
130 The last digit indicates the connection status (0: not connected, 1: connected).
131 
132 #### Initial setup
133 The initial setup needs to be done once for a new device or to register for a new SIM card.<br>
134 It is possible, that some of these commands are automatically set by the network provider, or that there are some commands missing because they were set automatically, by our network provider. This is to be treated as a guide and not as step-by-step instructions.<br>
135 1. Set up APN for GPRS connection profile "0" (first digit). All following commands use the default profile "0". Up to 7 profiles (0-6) can be configured at the same time.<br>
136 `AT+UPSD=0,1,"giffgaff.com"\r`<br>
137 1. Dynamic IP address assignment <br>
138 `AT+UPSD=0,7,"0.0.0.0"\r`<br>
139 1. Save the configuration in NVM: <br>
140 `AT+UPSDA=0,1\r` <br>
141 
142 #### Activate PDP
143 Every time after start-up, the internet connection has to be separately activated. While all other connections should happen automatically, this needs to be done explicitly.
144 1. Activate GPRS connection (recommended to do this in +CGATT:1)<br>
145 `AT+UPSDA=0,3\r` <br>
146 This command could take a while to perform (>1s in tests). The status OK is returned afterwards. In case of an error, there is not result returned (ERROR directly after the echo). <br>
147 1. Check the connection status <br>
148 `AT+UPSND=0,8\r` --> `+UPSND: 0,8,1\r` <br>
149 This command should be regularly performed, if internet connection is necessary.
150 1. Check if a IP address was assigned <br>
151 `AT+UPSND=0,0\r` --> `+UPSND: 0,0,"86.111.173.102"\r`
152

154 ## Sending a text message without storing it:
155 1. Setting the message format to text mode (PDU mode: 0) <br>
156 `AT+CMGF=1\r`
157 1. Specify the number to send the message to. <br>
158 `AT+CMGS="nbr"\r` <br>
159 The DCE will send the echo followed by ">" (on a new line)
160 1. Send the message as a string. <br>
161 `There can be line feeds, or whatever one likes :). However, with every \r, the SARA module will send the echo`
162 1. Send the message immediately<br>
163 `<ctrl-z>` <br>
164 Alternatively `<ESC>` discards the message. <br>
165 Only now will the SARA module send the status (OK, ERROR).

167 ## Getting the CellLocate Location:
168 First, the CellLocate server needs to be configured, this should be done separately from the main program. The setup needs a token, this should not be hardcoded into the software. (For more info see [CellLocate](#celllocate).)
169 
170 This command checks the possible settings:<br>
171 `AT+UGSRV=?\r` <br>
172 `+UGSRV: "srv1","srv2","token",(1,2,3,5,7,10,14),(1-5),(1-3),(1,64,65),(0-2),(0-15)\r` <br>
173 * The first two strings (note: they sent including the quotation marks!) specify the primary and secondary servers.
174 * The third one is the authentification token.
175 * The next value is the number of days the offline data should be valid for. Allowed values are shown above, the default value is 14.
176 * The next value is the number of weeks the offline data should be valid for. The range is from 1-4, the default value is 4.
177 * The next value is the resolution of the offline data in days. 1,2 and 3 are allowed values.
178 * The next value is the GNSS type: <br>
179  * 1: GPS <br>
180  * 64: GLONASS
181 <br>Default is 65 (both)
182 * The next value is the mode of the AssistNow Online data management.<br>
183  * 0: AssistNow Online data are downloaded at GNSS receiver power up<br>
184  * 1: AssistNow Online data automatically kept alive<br>
185  * 2: manual AssistNow Online data download<br>
186 * The next value is the bitmask for the desired data types for the online aiding. <br>
187  * 0: time<br>
188  * 1: position<br>
189  * 2: ephemeris<br>
190  * 4: almanac<br>
191  * 8: auxiliary<br>
192  * 16: ephemeris of satellites which are likely to be visible from the position estimated by the current registered network. This flag has no effect if the ephemeris flag is set to 0.<br>
193 The default value is 15.<br>
194 For a detailed description about all the parameters, see the [AT Command Manual](https://www.u-blox.com/en/docs/UBX-13002752)
195

201 ### Get the Location from the CellLocate server:
202 #### Possible configuration
203 To configure the CellLocate sensor, it is possible to configure what information the device reports to the server. There are two options:<br> (from the [Position Application Note](https://www.u-blox.com/sites/default/files/GNSS-Implementation_AppNote_%28UBX-13001849%29.pdf))
204 * ânormal scanâ: the cellular module only reports the parameters of the neighbouring visible cells designated by the network operator, which are normally collected by the module during its ânetworkâ activity. This configuration is suitable for a quick, rough update of location
205 * âdeep scanâ: the cellular module scans and reports all visible cells. This gives not only the parameters of the visible cells of the serving network operator but also the cells of all other available network operators, thus improving location accuracy. Although this takes a bit longer time (approx 30 s to 2 minutes is needed to perform a deep scan), and requires more communication traffic, as more data is sent to the server (9 bytes for each cell), the coverage, reliability and accuracy are all substantially better if hybrid positioning is preconfigured to operate in the âdeep scanâ mode of operation
206 
207 This setting can b made with the following command: <br>
208 `AT+ULOCCELL=<scan_mode>\r` <br>
209 * 0: normal scan<br>
210 * 1: deep scan<br>
211 
212 #### Request Location
213 The request location command specifies what type of location is requested, what response should be sent, timeouts and more: <br>
214 
215 `AT+ULOC=<mode>,<sensor>,<response_type>,<timeout>,<accuracy>[,<num_hypothesis>]\r`<br>
216 * <mode>: always needs to be 2, the other values are reserved<br>
217 * <sensor>: What type of sensor to use (bitmask): <br>
218  * 0: use last fix in internal database<br>
219  * 1: GNSS receiver (not included in our setup)<br>
220  * 2: CellLocate<br>
221 * <response_type> <br>
222  * 0: use last fix in internal database<br>
223  * 1: GNSS receiver (not included in our setup)<br>
224  * 2: CellLocate<br>
225 * <timeout>: in seconds from 1 to 999
226 * <accuracy>: in meters from 1 to 999999
227 * <num_hypothesis>: (optional) maximal desired responses from CellLocate (max. 16).
228 
229 Example: <br>
230 `AT+ULOC=2,2,0,60,500`<br>
231 This command will be echoed and answered quickly by the SARA module. The answer is only the status. Requesting the location can take a long time (see timeouts above), so the result is sent as a URC as soon as it is ready.
232 
233 The response will have the following structure (for the response type "0"):<br>
234 `+UULOC: <date>,<time>,<lat>,<long>,<alt>,<uncertainty>\r`<br>
235 `<uncertainty>` is the estimated 50% confidence level error in meters. <br>
236 
237 A possible response for response type "0":<br>
238 `+UULOC: `02/04/2020,21:06:15.000,55.8367527,-42209088,0,248\r` <br>
239 (The alert reader will be able to check where the PI was located during development.)
