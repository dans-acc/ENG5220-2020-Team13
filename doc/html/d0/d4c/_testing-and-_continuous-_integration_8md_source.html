1 # Testing and Continuous Integration
2 ## Continuous Integration
3 To automatically check whether the project compiles correctly without having to carry out the CMake build generation step at every push to or pull from Git, the team decided to set up continuous integration (CI) for the project. This ensures correct operations and improves fault detection: by continuously and automatically testing every small change made on the program it is quicker and easier to discover faults, resulting in reduced flaw search and mend times, smaller feature backlog and better overall reliability.
4 <br>
5 The team chose TravisCI, a popular CI tool. It integrates with Github smoothly, and has a wide variety of settings available, allowing us to configure the run environment to match that of our specific Raspberry Pi. This ensures that if the project passes the TravisCI checks it will run properly on the actual hardware we use too, even if the hardware itself is not available for development temporarily. This also aids future contributions from users outside the maintaining team, as they can verify the correctness of their code without access to our specific device. It can be run for specific branches, and build options such as compilers, necessary packages, processing instructions can be set individually.
6 <br>
7 The project can be found here: [![build](https://travis-ci.com/itsBelinda/ENG5220-2020-Team13.svg?token=kxeDEXY2kiCv968c3pfm&branch=master)](https://travis-ci.com/itsBelinda/ENG5220-2020-Team13)

9 ### Setting up TravisCI
10 To enable TravisCI to work with a project, the following steps need to be taken:
11 * Go to https://travis-ci.com/signin
12 * Choose "Sign in with Github"
13 * Allow everything and enter your password until you're redirected to your Github profile
14 * There click "Activate" under the Github Apps Integration
15 * There you'll be presented with a list of repositories you can allow TravisCI to be installed for - it's set to all repos by default, so select "Only select repositories", and select the project repository, then click approve and install
16 * Once it's done you'll see a list of Github repos on your TravisCI profile
17 * After this you only need to add a travis configuration file to the repository, and it is set.
18 * To enable TravisCI to be run at every push and/or pull, click "More Options" and "Settings" and choose the appropriate settings

20 ### The TravisCI configuration file
21 For TravisCI to detect a project (and branch) a file needs to be added to the root folder. The `.travis.yml` file needs to specify the language, the operating system, the operating system distribution, the compiler, any required packages that need to be installed, and any build commands. In case of using it with a code coverage tool, such as Codecov, the instructions to upload the code to Codecov’s servers need to be specified, too.
22 <br>
23 The `.travis.yml` file for this project is the following:
24 ```yaml
25 language: cpp
26 dist: bionic
27 sudo: false
28 
29 compiler:
30 # with CLang everything works fine, but gcov throws and out of memory error and doesn't upload to Codecov - might fix later if there's time
31 # - clang
32  - gcc
33 
34 os: linux
35 
36 addons:
37  apt:
38  packages: lcov
39 
40 install:
41  - sudo apt-get update
42  - sudo apt-get install libcpprest-dev
43 
44 before_script:
45  - cd software/BeeSafePI/build
46  - cmake ..
47 
48 script:
49  - make
50  - cd src/
51  - ./BeeSafePI
52  - cd ..
53  - ctest
54 
55 after_success:
56 # Create lcov report
57 - lcov --capture --directory . --output-file coverage.info
58 - lcov --remove coverage.info '/usr/*' --output-file coverage.info # filter system-files
59 - lcov --list coverage.info # debug info
60 # Uploading report to CodeCov
61 - bash <(curl -s https://codecov.io/bash) -f coverage.info || echo "Codecov did not collect coverage reports"
62 ```
63 In our case, it is important that the distribution is set to `bionic`, as otherwise the `libcpprest-dev` package is not detected. The `lcov` package also must be added as an addon, and the `libcpprest-dev` package must be installed before any of the build commands are run.
64 <br>
65 To match the build structure of the project, the cmake command must be called in the section `before_script`, while the make and ctest commands can be called in `script`.
66 <br>
67 If using a code coverages service, such as Codecov, any and all commands related to that must be called in `after_success`, as it requires a fully functional build before it can have a code coverage report created and uploaded to the service.
68 <br>
69 **Note** : An error encountered in this project was that when using CLang as the compiler, `gcov`, the parent package of `lcov`, a tool that helps create a code coverage report for the use of Codecov, throws an out of memory error, which causes the Travis build to run perfectly, however it does not upload to Codecov. When using gcc, the error is non-existent.

71 ## Unit testing
72 Due to the sensitive nature of transmitting the child’s location live, and being able to promptly respond when the child leaves their intended safe zones, it is of key importance that the program running on the Pi is tested thoroughly for any flaws and has the maximum possible uptime.
73 <br>
74 As the project is an embedded project with limited memory available - with even less if the planned future iterations are to go ahead with a Pi Zero - , it was important to consider what non-necessary packages and software were downloaded. Due to this, the testing was decided to be done in a low-level, traditional manner, without any testing frameworks such as Boost or GTest. CTest is CMake’s own test tool, and it only requires for the checks of specific pass or fail values to be returned from a test, making it fast and reliable for an embedded project. By default these values are 0 for pass and 1 for fail, however this was changed to specific strings in this project for better readability.
75 <br>
76 First, the CMake configuration needed to be edited to allow for the inclusion of CTest, by adding
77 ```
78 enable_testing()
79 add_subdirectory(tests)
80 ```
81 to the top level CMakeLists.txt file, as it allows the detection of test files and they can then be built with the project. In the tests directory, which is on the same level as the source file directory, a CMakeLists.txt file needed to be created to encapsulate the tests commands separately. As with unit tests code components need to be tested individually and to the lowest level, separate test files were created for each source class, making debugging easier. This, however resulted in over a dozen files that needed to be created. To avoid needing to edit the CMakeLists.txt file with every file added or removed, a generic script was added to it that parses every file with a specific name (all test files follow the naming convention of “test_” and name of the source class it tests), following this guide from Skand Hurkat: https://skandhurkat.com/post/intro-to-ctest/ .
82 ```cmake
83 set(EXECUTABLE_OUTPUT_PATH ${PROJECT_BINARY_DIR}/tests)
84 set(CTEST_BINARY_DIRECTORY ${PROJECT_BINARY_DIR}/tests)
85 
86 file(GLOB files "test_*.cpp")
87 
88 foreach(file ${files})
89  string(REGEX REPLACE "(^.*/|\\.[^.]*$)" "" file_without_ext ${file})
90  add_executable(${file_without_ext} ${file})
91  target_link_libraries(${file_without_ext} ${PROJECT_LIBS} PRIVATE cpprestsdk::cpprest)
92  add_test(${file_without_ext} ${file_without_ext})
93  set_tests_properties(${file_without_ext}
94  PROPERTIES
95  PASS_REGULAR_EXPRESSION "Test passed")
96  set_tests_properties(${file_without_ext}
97  PROPERTIES
98  FAIL_REGULAR_EXPRESSION "(Exception|Test failed)")
99  set_tests_properties(${file_without_ext}
100  PROPERTIES
101  TIMEOUT 120)
102 endforeach()
103 ```

105 ## Testing improvements
106 To verify the thoroughness of the tests, it is important to ensure as much of the software is covered by the tests as possible. A metric of this is using code coverage tools which run the tests and note what parts of the code are covered by checking what methods are called and what conditionals are chosen during the running of tests. Therefore, the project was released on one of the most popular code coverage services, which integrates with Github and TravisCI conveniently: Codecov.io.
